# 团队协作边界

## 模块负责人建议

| 模块 | 负责人 | 允许主要修改 | 禁止事项 |
| --- | --- | --- | --- |
| 前端 UI | UI 负责人 | `src/styles.css`, `src/components/`, `public/assets/` | 不改接口、状态机、路由 |
| 后端 API | 后端队友 A | `backend/`, 后续 API 文档 | 不改前端视觉组件 |
| 数据库/持久化 | 后端队友 B | `backend/`, 数据库迁移和模型说明 | 不改 mock 数据结构来绕过真实持久化 |
| AI/skill 接入 | 后端队友 C | `backend/`, AI 编排说明 | 不把 AI 调用写入 UI 组件 |
| 部署/CI | 队友 D | 部署脚本、CI 文档、环境变量说明 | 不硬编码 secret |

## AI 输入时请明确范围

推荐提示词格式：

```text
我负责【模块名】。
只允许修改【文件/目录】。
禁止修改【文件/目录/业务流程】。
目标是【具体任务】。
完成后必须运行【验证命令】。
```

## 前端 UI 保护规则

- 不删除已有功能按钮。
- 不改函数名、props 名、数据字段名。
- 不把业务逻辑写进 UI 组件。
- 不重构目录。
- 不删除路由。
- 不改变首页到历史足迹的主流程。

## 后端接入保护规则

- 通过 service 层对接前端，不让 UI 直接依赖后端 SDK。
- 新接口必须记录入参、返回结构、错误格式。
- 数据库字段变更必须先写迁移说明。
- AI/skill 能力先在后端封装，再由前端通过既有 service 调用。
